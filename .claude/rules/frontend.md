## Стиль кода

```
// ─── {NAME} ───
```

## JSDoc

Документация пишется на английском. Стиль — максимально краткий и по делу.

### Общие правила

- **Язык** — английский. Русские inline-комментарии при правке файла переводить на английский.
- **Запрещённые теги** — `@param`, `@returns`, `@throws`, `@template`. Описание параметров идёт через JSDoc на полях интерфейсов/inline options-объектов, а не в сигнатуре функции.
- **`@default`** — только если у пропса реально есть default в `defineProps` / `withDefaults`. Нет default — нет `@default`.
- **`@example`** — только когда использование не тривиально.
- **Не упоминать потребителей** — запрещено писать «used by X», «consumed by calendar/date picker», и т.п. Документация описывает саму сущность, а не где она применяется.
- **Без оправданий и мотивационных придаточных** — не писать «so misuse surfaces immediately», «via `defu`», «narrowed so consumers do not have to cast». Только факт.
- **Типы описываются одной строкой**, если этого достаточно. Поля интерфейсов — короткие однострочные описания.
- **Private-функции не документируются** — вложенные хелперы, функции в `lib/`, не-экспортируемые утилиты оставляй без JSDoc.
- **Очевидные утилиты не документируются** — если имя функции и сигнатура полностью раскрывают смысл (`isSameMonth`, `getWeekNumber`), JSDoc не нужен вовсе, либо одна строка максимум.
- **Не переписывать хорошую документацию** — если существующий JSDoc уже корректен и в стиле проекта, оставляй как есть. Правь только при ошибках или несостыковках.

### Глубина по категориям

- **Composables** — конкретное описание поведения (1–3 предложения), поля возвращаемого интерфейса с короткими JSDoc.
- **Props-поля** — краткое описание + `@default` (если есть). Документировать только те, что ещё не задокументированы.
- **Поля интерфейсов и типов (включая экспортируемые)** — да, короткими однострочниками.
- **Утилиты** — одна-две фразы по сути. Если поведение неочевидно (например, расширение falsy на пустые массивы, обработка CSS-единиц), кратко объяснить суть.

### Эталон минималистичного стиля

```ts
/** Check if two dates are in the same week */
export function isSameWeek(...) { ... }
```

Вот такой уровень краткости — предпочтительный дефолт для простых утилит.

## Структура компонентов

Каждый компонент в `src/runtime/components/<name>/` следует единой схеме:

```
components/<name>/
├── index.ts                  — публичные экспорты (типы + composables)
├── <name>.vue                — корневой компонент
├── <name>-<part>.vue         — подкомпоненты, если их мало (≤ 4)
├── ui/                       — подкомпоненты, если их много (calendar, combobox, table)
├── lib/                      — внутренняя логика: contexts, composables, helpers
│   ├── context.ts
│   └── use-*.ts
├── lib.ts                    — допустимо вместо папки `lib/`, если реально один файл
├── types.ts                  — типы, не привязанные к .vue (при необходимости)
└── <name>.module.css         — стили (если вынесены)
```

### Правила

- **Типы компонента** — только `types.ts` (не `model.ts`, не `.d.ts`). Исключение: `table.d.ts` с `declare module` для расширения чужих модулей.
- **`lib.ts` vs `lib/`** — если внутренней логики один файл, используй плоский `lib.ts`. Как только появляется второй файл — папка `lib/`.
- **Никаких папок с единственным `index.ts`** внутри компонента (например, `types/index.ts` с одним файлом → просто `types.ts`).
- **Префикс `_`** в названии папки разрешён **только** для папок, содержащих `.vue` компоненты, которые не должны быть доступны пользователю напрямую (например, `loader/_loaders/`, `modals/_confirm-modal/`, `tree/_ui/`). Для `lib/`, `ui/`, `types/` префикс не используется — они и так приватны по соглашению (не экспортируются из `index.ts`).

### `components/<name>/index.ts`

Формат:
```ts
// Типы корневого и подкомпонентов
export type * from './<name>-<part>.vue'
export type * from './<name>.vue'
// Публичные composables / утилиты из lib
export { useXxxState } from './lib/context'
// Локальные типы
export type * from './types'
```

### `components/index.ts` (корневой)

Для `.vue`-файлов, лежащих плоско в `components/`, **всегда** используй `export type * from './foo.vue'`, а не `export * from './foo.vue'`. Компоненты регистрируются Nuxt-модулем, поэтому рантайм-экспорт не нужен — только типы.

Для компонентов-директорий (имеющих свой `index.ts`) используй `export * from './<name>'`, так как их `index.ts` уже сам управляет тем, что публично (composables — рантайм, `.vue` — только типы).
